Architecture

Gumi is a Go modular monolith that sits between your application and your local inference engine. Every request flows through the Pipeline Engine, which orchestrates a series of specialized sub-engines.

Engines

Gateway Engine

Handles HTTP routing, authentication, request ID generation, and provider resolution. Stays intentionally thin — it forwards inference requests to the Pipeline Engine.

Pipeline Engine

The central orchestrator. Dispatches requests to the correct mode pipeline (direct, lightweight, stabilized, structured, or agent) and coordinates all sub-engines. Manages retry logic with exponential backoff.

Context Engine

Assembles session context, memory facts, and conversation history. Handles token estimation, message normalization, and sliding-window context compaction for long conversations.

Prompt Engine

Builds the final system prompt by combining base instructions, context blocks, format instructions, model profile instructions, and tool-calling hints. Produces a telemetry report of what was added.

Guard Engine

Pre-generation safety checks: anti-loop detection, context overflow prevention, and structured output enforcement. Runs before the provider call to prevent known failure modes.

Tool Shim Engine

Converts native OpenAI tool definitions into prompt-based instructions for models that don't reliably support native tool calling. Parses JSON tool calls from model output and validates them against the tool schemas.

Validation Engine

Post-generation checks: JSON validity, required keys, empty response detection, and schema compliance. Produces structured reports for telemetry and retry decisions.

Repair Engine

Attempts to fix common model output issues: broken JSON (missing braces, trailing commas), markdown-wrapped JSON, and repeated content. Reports success/failure for telemetry.

Memory Engine

Zero-VRAM persistent memory using SQLite. Stores facts (key-value with TTL and confidence), episode summaries (compressed step histories), and model-fit data (per-model performance tracking). Injects relevant facts into the system prompt before generation.

Instruction-Following Assist

Auto-detects 14 constraint types from user prompts (sentence count, word limits, forbidden words, JSON, line count, bullets, etc.) and injects explicit numbered reminders. Validates responses post-generation with automatic retry on failure.

Agentic Coding Router

Classifies each agent step by difficulty (trivial → novel) and task type (fix, refactor, feature, test, review, docs, search, plan) using structural heuristics. Selects the optimal model from available profiles. Re-evaluated each agent step.

Storage

Gumi uses SQLite for local storage with two databases:

  • Telemetry database (~/.gumi/gumi.db) — request metadata, pipeline events, validation reports, repair reports, provider health. Retained for 14 days by default. Full prompts and responses are never stored.
  • Memory database (~/.gumi/memory.db) — facts, episode summaries, model-fit data. Optional, opt-in via config.

Provider adapters

Gumi supports three local providers out of the box:

  • Ollama — http://localhost:11434
  • LM Studio — http://localhost:1234/v1
  • OpenAI-compatible local server — http://localhost:8000/v1

Provider adapters are intentionally thin: they handle URL routing, request/response translation, and error normalization. All intelligence lives in the pipeline engines.
